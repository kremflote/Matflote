# Web Design Ruleset

Single source of truth for designing the webpage in Figma and later implementing it in code.

## 1. Core design direction

The design should feel clean, friendly, readable, and structured. Use soft spacing, rounded corners, clear hierarchy, and a consistent 12-column layout.

Avoid random values. Reuse the values defined here unless there is a strong design reason not to.

---

## 2. Typography

### Font families

```
defaultFont: Nunito
heroFont: Segoe Print
logoFont: Ubuntu Bold 700
```

### Font usage

```
Segoe Print:
Use only for hero headings or major landing-page statements.

Ubuntu Bold 700:
Use only for the Fløte logo/brand mark.

Nunito:
Use for all normal interface text, body text, buttons, labels, cards, forms, navigation, and section headings.
```

Do not overuse the hero font. It should create contrast, not become the whole visual identity.

### Nunito weights

```
Nunito Light 300:
Use sparingly for subtle supporting text.

Nunito Regular 400:
Default body text.

Nunito SemiBold 600:
Buttons, labels, navigation, small headings.

Nunito Bold 700:
Card titles, section headings, important UI text.

Nunito ExtraBold / Black:
Use rarely for strong emphasis only.
```

### Type scale

```
Hero heading: 64px / Segoe Print
H1 page heading: 56px / Nunito Bold
H2 section heading: 40px / Nunito Bold
H3 card heading: 24px / Nunito Bold or SemiBold
Body text: 16px / Nunito Regular
Large body text: 18px / Nunito Regular
Small text: 14px / Nunito Regular
Caption: 12px / Nunito Regular
```

### Line height

```
Hero heading: 110%
Headings: 115–120%
Body text: 150%
Small text: 140–150%
```

### Text width

```
Paragraph max-width: 600–720px
Hero text max-width: 500–640px
Form max-width: 480–640px
```

Long text should never stretch across the full 12-column grid.

---

## 3. Grid system

Use a 12-column grid on desktop.

### Desktop

```
Frame width: 1440px
Content max-width: 1200px
Left/right margin: 120px
Columns: 12
Gutter: 24px
```

### Smaller desktop / laptop

```
Frame width: 1280px
Content max-width: 1120px
Left/right margin: 80px
Columns: 12
Gutter: 24px
```

### Mobile

```
Frame width: 390px
Columns: 4
Margin: 16px
Gutter: 16px
```

### Column usage

```
Full-width content block: 12 columns
Large text block + image: 6 columns + 6 columns
Wide feature card: 6 columns
Normal card in 3-card row: 4 columns
Small card in 4-card row: 3 columns
Form container: 5–6 columns
Text-heavy content: 6–8 columns
```

---

## 4. Spacing system

Use an 8px spacing system.

```
4px  = tiny detail spacing
8px  = small spacing
12px = compact meal planner grid spacing
16px = default spacing
24px = medium spacing
32px = large spacing
48px = small section spacing
64px = normal section spacing
96px = hero / major section spacing
```

Allowed spacing values:

```
4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 / 96
```

Do not use values like 13px, 27px, 35px, or 70px unless solving a very specific visual issue.

Use 12px only for compact repeated planner grids where 16px feels too loose, such as the meal calendar cell gaps.

---

### Navbar


### Hero section

```

```

### Normal section

```

```

```

```

### Section title spacing

```
Gap between section heading and content: 32px

```

## 6. Components

### Buttons

```
Default height: 48px
Small height: 36px
Large height: 56px
Horizontal padding: 24px
Border radius: 6px
Font: Nunito SemiBold 600
Text size: 16px
```

Use the small 36px height for compact app toolbars and dense control rows, such as the cookbook browser header buttons and the week planner controls. Related controls in the same first row should share this height.

### Input fields

```
Height: 48px
Horizontal padding: 16px
Border radius: 6px
Font: Nunito Regular 400
Text size: 16px
Gap between label and input: 8px
Gap between fields: 16px
```

### Textareas

```
Height: 120–160px
Padding: 16px
Border radius: 6px
```

### Cards

Cards should usually have automatic height based on content.

```
Padding: 24px
Border radius: 6px
Internal gap: 16px
Image height: 220–260px
Card title: 20–24px
Card text: 16px
Button height: 48px
```

Do not define cards mainly by fixed height. Define width, padding, gap, image height, and typography. Let the content determine the final height.

### Recommended card layout

```
Food/dish cards:
3 cards per row on desktop
Each card uses 4 columns
Image height: 220–260px
Card padding: 24px
Gap between cards: 24px
```

---

## 7. Border radius

Use only these radius values.

```
Small elements / badges: 6px
Buttons: 6px
Inputs: 6px
Cards: 6px
Images: 6px
```

Do not mix many different corner radius values.

---

## 8. Elevation / shadows

Use shadows sparingly to clarify interactive surfaces and layered UI. Do not invent one-off shadow values in component files; use the centralized shadow classes from `frontend/src/styles/appStyles.ts`.

```
Subtle shadow:
Use for compact controls, search fields, filters, thumbnails, and small interactive surfaces.

Raised shadow:
Use for persistent page structure such as the main header.

Overlay shadow:
Use for popovers, dropdowns, and modal panels.
```

Shadows should support hierarchy without becoming decorative. If a surface already has strong color contrast and does not need depth, do not add a shadow.

---

## 9. Shared application patterns

Use centralized frontend style objects and shared browser components before adding page-local styling.

```
Global design/style tokens:
frontend/src/styles/appStyles.ts

Cookbook/browser shared components and filter styles:
frontend/src/components/recipeBrowser/

Planner picker-specific layout:
frontend/src/components/plannerRecipePicker/
```

Shared browser/filter UI should be reused across Cookbook and Planner when the interaction is the same:

```
Search fields
Filter icon buttons
Ingredient filter popovers
Active removable filter chips
Checkbox filter groups
Disabled/greyed filter options
```

Page-specific files may define layout wrappers, but not duplicate the same browser controls.

### Confirmation dialogs

Use the shared `ConfirmationDialog` component for destructive or bulk actions. Do not use `window.confirm` for app workflows.

Current uses:

```
Cookbook remove recipe / ingredient / brand
Planner clear week / month
Planner generate week / month
Planner remove meal
```

Confirmation dialogs should support Escape-to-close, focus the cancel action first, and use the centralized overlay shadow.

---

## 10. Icons

```
Small icons: 16px
Default icons: 24px
Large icons: 32px
```

Icons should align visually with text and buttons.


# Features

- Weekplanner
	- 7 day week overview. Each day you can fill in 3 (will be configurable later) slots:
		- Recipe
			- can have up to 4 "sides", meaning any recipe that is not a dish
	- Planner top controls:
		- Use compact 36px controls, matching Cookbook first-row control height
		- Year sits above week/month label as a non-layout overlay
		- Week/month toggle uses the same visual language as Cookbook toggle tabs
	- Week view meal card:
		- Recipe image fills the full left half of the meal container
		- Image has rounded top-left and bottom-left corners
		- Right half contains main dish title at top-left and side capsules below
		- Breakfast/lunch/dinner labels are currently hidden
	- Month view:
		- Each day shows clickable breakfast, lunch, and dinner rows
		- Existing meals open the edit flow
		- Empty rows open the same meal picker to add a meal
	- Meal picker flow:
		- Opening a slot always starts on main dish selection
		- Clicking a main dish highlights it; the user confirms with the bottom-left action before choosing sides
		- Supplement selection allows up to 6 sides/sauces/dips/spice mixes/salads
		- The bottom selected section shows the main dish thumbnail and selected side capsules
		- Save is disabled while the main dish highlight is unconfirmed
		- Remove meal is an explicit destructive action in edit mode
	- Week generator button
		- Current implementation is intentionally simple: fill empty slots only with a random dish and optional side
		- Future implementation should generate based on protein, cuisine, tags, repetition, and slot type
	- Prep helper
		- Generate a list of all vegetables you need to prepare for week
		- Automatically generate vikunja shopping list
- Cookbook
	- Create dishes, desserts, sauces, dips, marinades, spice mixes
	- Add ingredients
	- Overview of all recipes and ingredients
	- Browser filters, ingredient popover, active filter chips, and checkbox filter groups are shared patterns used by Planner as well
- Settings page
	- Configure amount of meals one can plan each day
	- other settings

# Data Notes

## Ingredient tags

`IngredientTag.LeafyGreen` exists in both backend and frontend enum/type definitions.

Existing database ingredients are tagged through the `TagLeafyGreenIngredients` migration when their names match common leafy greens such as spinach, kale, lettuce, arugula, rocket, cabbage, or chard.

Future seeded leafy ingredients should normally receive both:

```
Vegetable
LeafyGreen
```

Recipes do not store `LeafyGreen` directly. Planner filters infer it from the tags on recipe ingredients.
