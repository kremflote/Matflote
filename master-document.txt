# Web Design Ruleset

Single source of truth for designing the webpage in Figma and later implementing it in code.

## 1. Core design direction

The design should feel clean, friendly, readable, and structured. Use soft spacing, rounded corners, clear hierarchy, and a consistent 12-column layout.

Avoid random values. Reuse the values defined here unless there is a strong design reason not to.

---

## 2. Typography

### Font families

```
defaultFont: Nunito
heroFont: Segoe Print
logoFont: Ubuntu Bold 700
```

### Font usage

```
Segoe Print:
Use only for hero headings or major landing-page statements.

Ubuntu Bold 700:
Use only for the MATFLOTE logo/brand mark.

Nunito:
Use for all normal interface text, body text, buttons, labels, cards, forms, navigation, and section headings.
```

Do not overuse the hero font. It should create contrast, not become the whole visual identity.

### Nunito weights

```
Nunito Light 300:
Use sparingly for subtle supporting text.

Nunito Regular 400:
Default body text.

Nunito SemiBold 600:
Buttons, labels, navigation, small headings.

Nunito Bold 700:
Card titles, section headings, important UI text.

Nunito ExtraBold / Black:
Use rarely for strong emphasis only.
```

### Type scale

```
Hero heading: 64px / Segoe Print
H1 page heading: 56px / Nunito Bold
H2 section heading: 40px / Nunito Bold
H3 card heading: 24px / Nunito Bold or SemiBold
Body text: 16px / Nunito Regular
Large body text: 18px / Nunito Regular
Small text: 14px / Nunito Regular
Caption: 12px / Nunito Regular
```

### Line height

```
Hero heading: 110%
Headings: 115–120%
Body text: 150%
Small text: 140–150%
```

### Text width

```
Paragraph max-width: 600–720px
Hero text max-width: 500–640px
Form max-width: 480–640px
```

Long text should never stretch across the full 12-column grid.

## 4. Spacing system

Use an 8px spacing system.

```
4px  = tiny detail spacing
8px  = small spacing
12px = compact meal planner grid spacing
16px = default spacing
24px = medium spacing
32px = large spacing
48px = small section spacing
64px = normal section spacing
96px = hero / major section spacing
```

Allowed spacing values:

```
4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 / 96
```

Do not use values like 13px, 27px, 35px, or 70px unless solving a very specific visual issue.

Use 12px only for compact repeated planner grids where 16px feels too loose, such as the meal calendar cell gaps.

```
Outer modal max height: 88vh
Inner form/info body max height: 56vh
Mobile inner body max height: 52vh

```

Use the outer max height on the modal frame and the inner max height on the scrollable form/info body container in recipe create/edit and recipe/ingredient info windows. ### Cards

```

```

Do not define cards mainly by fixed height. Define width, padding, gap, image height, and typography. Let the content determine the final height.

### Recommended card layout

```

## 8. Elevation / shadows

Use shadows sparingly to clarify interactive surfaces and layered UI. Do not invent one-off shadow values in component files; use the centralized shadow classes from `frontend/src/styles/appStyles.ts`.

```
Subtle shadow:
Use for compact controls, search fields, filters, thumbnails, and small interactive surfaces.

Raised shadow:
Use for persistent page structure such as the main header.

Overlay shadow:
Use for popovers, dropdowns, and modal panels.
```

Shadows should support hierarchy without becoming decorative. If a surface already has strong color contrast and does not need depth, do not add a shadow.

## 9. Shared application patterns

Use centralized frontend style objects and shared browser components before adding page-local styling.

```

### Mobile layout rules

Design mobile layouts explicitly for phone widths around 360-430px, then verify tablet and desktop remain intact.

Use the Tailwind `max-md` breakpoint for tablet/phone layout shifts and `max-sm` for phone-specific tightening. Mobile values should live in centralized style helpers before they are used in page components.

Current centralized mobile helpers live in `responsiveClasses` in `frontend/src/styles/appStyles.ts`.

Mobile layout defaults:

```
Page horizontal width: layoutClasses.contentWidth
Page vertical padding: 32px on mobile, 24px on small phones
Modal viewport inset: 16px normally, 12px on small phones
Modal max height: calc(100vh - 24px) on small phones
Touch targets: at least 40px on small phones where controls are compact
```

Mobile behavior rules:

- Header keeps logo/theme at the top on mobile, while primary navigation moves to a fixed bottom bar
- Header logo shrinks on small phones, but remains the first visual brand signal
- Header navigation buttons become bottom tab controls and share available width
- Bottom navigation uses dedicated `siteColorClasses[theme].bottomNav` and mobile-strength nav button colors; do not rely on transparent header button styling for the fixed mobile bar
- Mobile pages need enough bottom padding that fixed bottom navigation does not cover final content
- Toolbars may wrap, but related controls should stay visually grouped
- Page grids should collapse to one column before causing horizontal scrolling
- Desktop data grids and calendars may remain horizontally scrollable only when a mobile-specific replacement has not yet been implemented
- Modals should keep headers and actions reachable, with only their body content scrolling when practical
- Forms should become single-column on mobile and keep labels close to their fields
- Avoid mobile-only one-off spacing values in components; add a token or helper first

Mobile audit status:

- Header/navigation: first responsive pass implemented
- Shared modal/backdrop helpers: first responsive pass implemented
- Cookbook: first responsive pass implemented for header actions, search/filter controls, category filter modal, result grids, detail rows, and modals. On mobile, the Add action becomes a round floating action button above the fixed bottom navigation. The create modal keeps its recipe/ingredient mode switch full-width and lowers the scrollable form body height on phone widths so the footer actions remain reachable. Detail modals use the same phone pattern: stable close control, scrollable detail body, and bottom edit/remove actions outside the scroll area.
- Planner week view: first responsive pass implemented; controls stack and days render as a vertical list on tablet/phone widths
- Planner top controls use two rows on desktop: the week/month counter row is centered above an action row. Action buttons take the 8-column action area, and the week/month toggle keeps the shared thumbnail control width. On mobile, keep the counter first and replace the desktop action row with two compact buttons below it: one opens a small planner tools dialog for Clear/Generate/Prep/Export, and one cycles Week/Month as a single button.
- Planner meal picker uses the shared browser pattern: search, Categories button, ingredient filter button, and active filter chips. Category filters open in a modal instead of occupying a permanent side rail.
- Planner week day sections are collapsible on mobile widths only. Desktop day labels are static; mobile day titles act as toggles and show a chevron-style state symbol beside the title. Mobile day headers also show three compact right-aligned meal preview squares for the day's slots; filled slots use the main dish image and empty slots use the planner empty-slot treatment.
- Planner month view: first responsive pass keeps the 7-column calendar horizontally scrollable on narrow screens; compact month interaction review remains a later design task
- Settings: pending mobile panel stacking and form review

Shared browser/filter UI should be reused across Cookbook and Planner when the interaction is the same:

```
Search fields
Filter icon buttons
Ingredient filter popovers
Active removable filter chips
Checkbox filter groups
Disabled/greyed filter options
```

Page-specific files may define layout wrappers, but not duplicate the same browser controls.

### Production-style local builds

The backend can serve the built frontend from `backend/wwwroot`, but those files are generated output. Refresh them through `scripts/deploy-frontend.ps1` before local production-style testing. Use `scripts/build-app.ps1` when both the frontend deployment and backend Release build should be refreshed together.

Keep user-uploaded images separate from generated frontend assets. Runtime images are served from `/images/*` through the backend image storage configuration, while generated frontend files under `backend/wwwroot` can be replaced by the deployment script.

### Docker deployment assumptions

Docker v1 is a single-household, single-server deployment. Phones, tablets, and computers sync by connecting to the same MATFLOTE server over LAN or a hosted address.

The Docker setup uses a separate nginx frontend container and backend API container. nginx provides one browser origin, serves the built frontend, and proxies `/api/*`, `/images/*`, and `/health` to the backend.

SQLite is accepted for this mode because one backend process owns the database file. The Docker database and uploaded images live in persistent volumes. PostgreSQL, accounts, households, auth, and offline multi-device sync are deferred future architecture decisions.

### Shopping list export assumptions

Shopping list export is provider-based. MATFLOTE first builds a neutral grocery list from planner meals, then passes it to the configured exporter. Vikunja is the first exporter, but other todo systems should be implemented as additional exporters rather than by coupling planner logic directly to a third-party API.

Provider config is server-side. Environment variables and appsettings files seed the first-run configuration, and the Settings page can store runtime overrides in the database. API tokens must not live in frontend code, committed source files, or generated static assets. The Settings UI must never show a saved token value; it may only show whether one is configured and accept a replacement token.

Vikunja export supports two task modes:

```
SingleTask:
One Vikunja task containing a checklist-style shopping list in the task description.

SeparateTasks:
One Vikunja task per ingredient. Task titles include ingredient name, brand when present, compact amount, and source meal names.
```

SingleTask should remain the default to avoid spamming the todo project.

The backend exposes a preview endpoint before export so the generated neutral grocery list can be inspected without calling external services. Export endpoints should build from the same neutral list instead of duplicating planner aggregation logic.

The Settings page may expose exporter options such as one checklist task versus separate ingredient tasks, plus a connection test. Connection tests must use the same server-side validation and token handling rules as real exports.

### Runtime verification

Use `scripts/smoke-test.ps1` as the first verification pass after backend/frontend integration changes. The smoke script should remain external-service-free. It may check local endpoints, temporary SQLite migration/startup behavior, runtime settings persistence, grocery preview generation, and image upload storage, but it should not require a real Vikunja token.

Use Docker checks for deployment readiness:

```
docker compose config
docker compose build
docker compose up -d --build
```

Fresh Docker runs should prove that backend startup migrations create or update the SQLite database, runtime settings can be persisted, and images remain in the persistent image volume after container restart.

### Starter data catalog

Starter ingredients and recipes should be shipped through `backend/SeedData/catalog.json`, not by editing a personal SQLite database.

Backend startup imports this catalog after EF migrations. The import is additive and idempotent:

- Brands and cuisines are created when missing
- Ingredients are matched by ingredient name plus optional brand name
- Recipes are matched by recipe name plus recipe type
- Existing matching records are not overwritten

The app exposes `GET /api/seed-catalog/export` so curated data can be created through the UI, exported to JSON, reviewed, and then committed as starter data.

The JSON catalog should use stable English enum identifiers for recipe type, recipe tags, ingredient tags, measurement units, dessert type, and ingredient preparation.

### Confirmation dialogs

Use the shared `ConfirmationDialog` component for destructive or bulk actions. Do not use `window.confirm` for app workflows.

Current uses:

```
Cookbook remove recipe / ingredient / brand
Planner clear week / month
Planner generate week / month
Planner remove meal
```

Confirmation dialogs should support Escape-to-close, focus the cancel action first, and use the centralized overlay shadow.

### Language support

Use a frontend localization layer for interface text. The active language is provided by `LanguageContext`, persisted in local storage, and selected from the Settings page.

Current supported interface languages:

```
English
Norwegian Bokmål
```

Backend enum values and frontend union keys should remain stable English identifiers. User-facing labels for recipe types, recipe tags, ingredient tags, meal slots, planner view modes, and measurement units should come from `frontend/src/i18n/index.ts`.

Lunch and Dinner remain backend-compatible recipe tag identifiers for older saved recipes, but they are not selectable in the recipe form. Breakfast is the only meal-slot-style recipe tag still exposed in the UI.

User-created content such as recipe names, ingredient names, cuisine names, instructions, and descriptions is not translated automatically.

# Features

- Weekplanner
	- 7 day week overview. Each day you can fill in 3 (will be configurable later) slots:
		- Recipe
			- can have up to 4 "sides", meaning any recipe that is not a dish
	- Planner top controls:
		- Use compact 36px controls, matching Cookbook first-row control height
		- Year sits above week/month label as a non-layout overlay
		- Week/month toggle uses the same visual language as Cookbook toggle tabs
	- Week view meal card:
		- Recipe image fills the full left half of the meal container
		- Image has rounded top-left and bottom-left corners
		- Right half contains main dish title at top-left and side capsules below
		- Breakfast/lunch/dinner labels are currently hidden
	- Month view:
		- Each day shows clickable breakfast, lunch, and dinner rows
		- Existing meals open the edit flow
		- Empty rows open the same meal picker to add a meal
	- Meal picker flow:
		- Opening a slot always starts on main dish selection
		- Clicking a main dish highlights it; the user confirms with the bottom-left action before choosing sides
		- Supplement selection allows up to 6 sides/sauces/dips/spice mixes/salads
		- Supplement selection excludes Breakfast-tagged recipes, but does not exclude legacy Lunch or Dinner tags
		- The bottom selected section shows the main dish thumbnail and selected side capsules
		- Save is disabled while the main dish highlight is unconfirmed
		- Remove meal is an explicit destructive action in edit mode
	- Week generator button
		- Current implementation is intentionally simple: fill empty slots only with a random dish and optional side
		- Future implementation should generate based on protein, cuisine, tags, repetition, and slot type
	- Prep helper
		- Week view includes a compact prep-helper button next to the other planner action buttons
		- The prep helper opens a modal listing produce-style ingredients needed for the current week
		- Produce-style ingredients are currently inferred from ingredient tags: Vegetable, Fruit, Herb, and LeafyGreen
		- Recipe ingredients store preparation as explicit enum data, separate from measurement unit
		- Measurement unit describes amount, such as gram, piece, or clove
		- Ingredient preparation describes shape/state, such as quartered, wedged, chopped, roughly chopped, finely chopped, diced, cubed, julienned, batons, sliced, minced, grated, shredded, or crushed
		- The prep helper uses explicit recipe-ingredient preparation first
		- If no explicit preparation is stored, prep actions may still be inferred from recipe instructions when they contain words such as chop, roughly chop, cube, dice, julienne, batons, slice, mince, grate, shred, crush, quarter, or wedge
		- If no prep action can be inferred, the ingredient is omitted from the prep helper
		- Automatically generate vikunja shopping list
- Cookbook
	- Create dishes, desserts, sauces, dips, marinades, spice mixes
	- Add ingredients
	- Overview of all recipes and ingredients
	- Browser filters, ingredient popover, active filter chips, and checkbox filter groups are shared patterns used by Planner as well
- Settings page
	- Choose interface language
	- Configure amount of meals one can plan each day
	- other settings

# Data Notes

## Ingredient tags

`IngredientTag.LeafyGreen` exists in both backend and frontend enum/type definitions.

Existing database ingredients are tagged through the `TagLeafyGreenIngredients` migration when their names match common leafy greens such as spinach, kale, lettuce, arugula, rocket, cabbage, or chard.

Future seeded leafy ingredients should normally receive both:

```
Vegetable
LeafyGreen
```

Recipes do not store `LeafyGreen` directly. Planner filters infer it from the tags on recipe ingredients.
